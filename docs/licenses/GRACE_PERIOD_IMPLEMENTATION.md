# Grace Period Implementation - 7-Day Write Block After Expiry

## Overview

Added a **7-day grace period** after license expiry. After the grace period ends, **writes are blocked** to the cluster. The grace period start time is:
1. Stored in the Raft-replicated `system.licenses` table
2. **Protected by a cryptographic signature** to prevent tampering
3. Automatically tracked by the compliance monitor

## Key Features

### ✅ Security: Tamper-Proof Grace Period
The grace period start timestamp is protected by a **signature** that covers:
- Original license data
- Grace period start timestamp

Formula: `signature = SHA256(public_key + license_data + ":" + grace_start_timestamp)`

**Why this works:**
- Users cannot modify the grace period timestamp without breaking the signature
- The signature is verified on every compliance check
- If tampering is detected, grace period is treated as expired

### ✅ Automatic Tracking
- Grace period starts automatically when license first detected as expired
- Tracked by the Raft leader node (no coordination needed)
- Persisted in Raft table (survives restarts and leader changes)

### ✅ Clear Warnings
Users see escalating warnings:
1. **License expired**: "Grace period active. Writes will be blocked in X days"
2. **Grace period ending soon**: "Writes will be blocked in 1 day"
3. **Grace period ended**: "Writes are BLOCKED. Renew license immediately"

## Security Analysis

### Attack Scenarios & Defenses

#### ❌ Attack: Modify grace_period_start_timestamp in database
**Defense**: Signature verification fails → grace period treated as expired → writes blocked

#### ❌ Attack: Delete grace period columns via CQL
**Defense**: Table not accessible via CQL, only through REST API

#### ❌ Attack: Rollback database to before grace period started
**Defense**: Compliance monitor re-detects expiry, sets grace period again

#### ❌ Attack: Replace entire license with older one
**Defense**: License signature covers expiry date, older license also expired

#### ❌ Attack: Modify system clock
**Defense**: Doesn't help - grace period based on stored timestamp, not current time

### Why This is Secure

1. **Signature covers both license AND grace start**
   - Can't change grace start without breaking signature
   - Can't reuse old license (also expired)

2. **Server-side generation**
   - Grace period timestamp generated by server, not client
   - Signature uses server's embedded public key (not accessible to users)

3. **Raft replication**
   - Grace period state replicated to all nodes
   - Consistent view across cluster

4. **Idempotent**
   - If grace period already started, doesn't restart
   - Safe to call multiple times

## User Experience Flow

### Day 0: License Expires
```
[Compliance Check]
→ License expired detected
→ Grace period starts NOW
→ Update system.licenses with grace_period_start_timestamp
→ Generate signature for (license_data + ":" + grace_start)
→ Log: "License expired. Starting 7-day grace period"
→ Cluster continues operating normally
```

### Days 1-6: Grace Period Active
```
[API Status Check]
GET /v2/license/status
→ {
    "status": "expired",
    "message": "License expired. Grace period active. Writes will be blocked in X days",
    "days_until_write_block": X
  }

[Cluster Behavior]
→ All operations work normally
→ Warnings logged every compliance check
→ Admins have time to renew
```

### Day 7+: Grace Period Ended
```
[Write Operation]
→ Check: grace_period_exceeded = true
→ Block write with error: "License expired. Grace period ended. Renew license."

[Read Operations]
→ Continue working normally
→ No impact on reads

[API Status Check]
GET /v2/license/status
→ {
    "status": "expired",
    "message": "License expired. Grace period ended. Writes are BLOCKED",
    "days_until_write_block": 0
  }
```

## Integration Points

### Where Write Blocking Happens
Writes should be blocked in the **write path** of ScyllaDB. Suggested locations:

1. **storage_proxy** - Before applying mutations
2. **CQL handler** - Before executing INSERT/UPDATE/DELETE
3. **API endpoints** - Before write operations

### Pseudocode for Write Check
```cpp
// In write path (e.g., storage_proxy::mutate)
if (license_service.is_write_blocked()) {
    throw std::runtime_error(
        "Writes are blocked. License expired and grace period ended. "
        "Contact ScyllaDB to renew your license.");
}
// ... proceed with write ...
```

### Compliance Monitor Integration
```cpp
// In compliance_monitor::do_check()
seastar::future<> compliance_monitor::do_check() {
    if (should_skip_check()) {
        co_return;
    }
    
    // Check and update grace period if needed
    co_await _license_service.local().check_and_update_grace_period();
    
    // ... rest of compliance check ...
}
```
